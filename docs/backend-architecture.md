# 后端架构设计

## 目标

这是一个公开的翻译协作平台，单个系统支持多个组织、多个项目。

每个项目只负责一条翻译线，例如：

- `en/jp/kr -> cn`
- `en -> cn`
- `cn -> en`

如果要做另一条翻译线，就新建一个项目，而不是在同一个项目里混用。

## 总体方案

先做“模块化单体”，不要一开始拆微服务。

原因很简单：

- 你的复杂度主要在权限、版本、导入导出、搜索筛选
- 这些功能强依赖同一套数据库事务
- 一开始拆服务只会把问题变成跨服务一致性问题

所以第一版建议：

- 后端：Go
- 路由：`chi`
- 数据库：PostgreSQL
- 缓存和任务队列：Redis
- SQL 管理：`sqlc + pgx`
- 接口风格：REST

## 模块划分

建议后端按下面这些模块组织：

- 用户与登录
- 组织
- 项目
- 项目成员
- 权限
- 语言配置
- 文档
- 文档标签
- 翻译条目
- 版本历史
- 术语库
- 翻译记忆
- 导入
- 导出
- 搜索与筛选
- 审计日志
- 平台多语言

虽然这些都在一个服务里，但代码和数据库都要按模块分清楚。

## 核心层级

推荐的业务层级如下：

```text
平台
  └─ 组织
       └─ 项目
            └─ 文档
                 └─ 翻译条目
```

说明：

- 组织用于管理多个项目
- 项目是权限、语言配置、术语库、翻译记忆库的隔离边界
- 文档对应导入导出的一个 JSON 文件
- 翻译条目对应文档中的一个 `key`

## 项目模型

每个项目固定以下配置：

- 所属组织
- 项目名称
- 项目代号
- 源语言列表
- 目标语言
- 是否公开
- 默认访客策略

例如：

```text
项目：Arknights-CN
源语言：en, jp, kr
目标语言：cn
```

## 文档模型

文档不是简单文件名，而是一个长期存在的业务对象。

建议文档至少包含：

- 文档路径，例如 `StoryData/S001B.json`
- 文档标题
- 文档标签列表
- 当前版本号
- 最后更新时间
- 最后更新人

文档标签的用途：

- 前端可显示为 label
- 可用于筛选
- 后续可用于权限规则
- 也可以不显示，只作为内部分类

## 翻译条目模型

每个条目在项目内以 `document + key` 唯一。

每个条目应包含：

- 所属文档
- key
- 多个原文
- 一个当前译文
- 当前状态
- 备注
- 版本号
- 最后编辑人
- 最后编辑时间

这里的“多个原文”是这个系统和一般 i18n 平台不一样的重点。建议内部结构就是：

```text
条目
  key = home.title
  source.en = ...
  source.jp = ...
  source.kr = ...
  target.cn = ...
```

## 状态流转

第一版保持简单：

- 未翻译
- 已翻译
- 已校对
- 已批准

角色能力：

- 翻译者：修改译文、提交翻译
- 校对者：修改译文、校对、批准
- 管理员：管理项目、成员、权限、导入导出

你已经明确说了第一版不需要“多个候选译文并排竞争”，所以这版设计为：

- 只有一个当前译文
- 每次修改都记入历史
- 校对者直接在当前译文上工作

## 版本管理

版本管理分两层就够了。

### 文档版本

每次导入文档时，生成一个新的文档版本。

这个版本保存：

- 这次导入对应的原始内容摘要
- 导入时间
- 导入人
- 版本号

### 条目修订历史

每次条目发生变化时，记录一条修订历史。

记录内容包括：

- 变更前译文
- 变更后译文
- 变更前状态
- 变更后状态
- 操作人
- 操作时间
- 变更说明

这样已经足够接近 MediaWiki 的“页面历史”体验，不需要 branch。

## 权限系统

不要把权限硬写死在角色里，应该做成“权限节点 + 角色 + 覆写 + 范围限制”。

### 四层结构

第一层：权限节点

第二层：角色

第三层：用户单独授权或拒绝

第四层：文档范围规则

### 角色只是预设

默认角色：

- owner
- admin
- reviewer
- translator
- guest

但这些角色本质上只是权限节点集合。

### 文档范围规则

你提的白名单/黑名单很重要，建议这样做：

- 支持按路径前缀限制
- 支持按文档标签限制
- 支持 allow 和 deny

例如：

- 某翻译者只能编辑 `StoryData/`
- 某用户不能编辑标签为 `locked` 的文档

第一版建议默认：

- 读权限和写权限分开
- 文档范围规则至少先作用于写权限
- 如果后面需要，也可以扩展到读权限

## 搜索与筛选

第一版先用 PostgreSQL 就够了，不急着单独上搜索引擎。

需要支持：

- 按项目筛选
- 按文档筛选
- 按标签筛选
- 按 key 搜索
- 按状态筛选
- 按原文搜索
- 按译文搜索
- 按负责人或最后编辑人筛选

如果后面数据量很大，再考虑额外搜索组件。

## 导入

导入单位是“单个文档 JSON”。

例如：

- `StoryData/S001.json`
- `StoryData/S002.json`

导入流程建议：

1. 上传 JSON
2. 校验格式
3. 校验语言配置是否匹配项目
4. 写入导入任务
5. 后台处理
6. 生成文档版本
7. 更新条目
8. 写入审计日志

导入不要直接同步阻塞请求，否则大文件会拖垮接口体验。

## 导出

导出单位是“整个项目”。

导出结果是：

- 一个 zip 包
- 里面是多个 JSON 文件
- 每个文档一个 JSON

导出流程建议：

1. 创建导出任务
2. 后台收集项目文档
3. 生成多个 JSON
4. 打包为 zip
5. 记录下载地址和过期时间

## 导出清理

既然你已经确认用 zip，就必须顺手处理清理策略。

建议第一版直接定死：

- 导出文件保留 24 小时
- 每小时清理一次过期导出文件
- 同时限制每个项目最多保留最近 20 个导出包

这样时间和数量双保险，不容易把磁盘堆满。

## 平台多语言

你要求支持 i18n，这里分两层：

### 平台界面多语言

平台自己的按钮、菜单、提示语要支持多语言。

例如：

- 简体中文
- 英文
- 日文

这个和翻译项目本身是两回事，不要混在一个表里。

### 翻译内容多语言

这是项目里的原文和译文数据。

这两套多语言要分开设计。

## 部署建议

第一版不追求花哨，建议：

- 一个 Go API 服务
- 一个 worker 进程
- 一个 PostgreSQL
- 一个 Redis
- 一个前端服务

本地开发和小规模部署都可以用 Docker Compose。
